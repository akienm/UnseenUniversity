# Chapter 5: The Platform — Unseen University

---

## Why a Platform

The skills and workflow described in the previous chapters require infrastructure to run on. Not elaborate infrastructure — but specific infrastructure. A place to store state. A way for components to communicate. A mechanism for devices to be added to and removed from the system without rewriting everything around them.

The Unseen University (UU) is that infrastructure. It is not an AI system. It is the substrate on which AI systems run. The distinction matters: an AI system makes decisions, processes information, and produces outputs. A platform provides the services those systems depend on — storage, messaging, logging, discovery, launch.

Building the AI on top of a platform rather than directly on infrastructure means the AI system can change without the platform changing, and the platform can improve without requiring changes to the AI. The two concerns are separated, and separation is the basis of maintainability.

---

## The Rack Model

UU organizes its components as rack-mounted devices. A device is any process that performs a defined function within the system: the AI reasoning engine (Igor), the knowledge management system (Librarian), the deterministic script runner (Scraps), the web server, the queue manager.

Each device has:
- A name and a rack slot
- A config file at `~/.UnseenUniversity/<DeviceName>/config.cfg`
- A log directory at `~/.UnseenUniversity/logs/<DeviceName>/`
- A health check that the rack can poll
- A defined interface for the services it provides

The rack model enforces a clean separation between "what a device does" and "how it fits into the system." A device that fails can be restarted independently. A new device can be added without modifying existing devices. The rack is the integration point, not the devices themselves.

## Discovery Over Configuration

One of the explicit design goals of UU is to eliminate required configuration. The system should discover what it needs rather than require the user to configure it.

The primary example is instance discovery. When Igor starts, it does not require an environment variable telling it which instance to use. It looks at `~/.UnseenUniversity/Igor-*/` and:

- If it finds one instance: uses it
- If it finds more than one: asks which to start
- If it finds none: runs the fresh-install wizard

The `--id` flag is available as an explicit override, but it is not required. The common case — one instance on one machine — requires no configuration at all.

The same principle applies throughout the platform. Devices discover each other via the rack manifest rather than via hardcoded addresses. Configuration that can be derived from the filesystem is derived rather than specified. Required environment variables are eliminated where possible.

This is not just convenience. It is the difference between a system that a new user can start without a setup guide and one that requires a half-hour of configuration before it runs.

---

## Config Over Environment Variables

Each device's configuration lives in `~/.UnseenUniversity/<DeviceName>/config.cfg`. All the parameters that used to be environment variables — API keys, database URLs, model preferences, feature flags — live in a single file that the device reads at boot.

Why this matters: when something goes wrong and you need to diagnose it, the first question is "what was the system configured to do?" With environment variables, the answer requires reconstructing the shell environment at the time of the failure — which may be gone. With a config file, the answer is in a single readable file that persists.

The config file is also the right boundary for human-readable documentation of the device's operating parameters. It can be committed to a private repo, diffed across environments, and reviewed as part of a debugging session. Environment variables cannot.

## The Communication Layer

Devices in the rack communicate through structured channels rather than direct calls. A channel is a named message queue that any device can write to and any device can subscribe to. The shared channel is where Igor, Librarian, and the Claude Code session exchange messages. Device-specific channels handle point-to-point communication.

The channel model has two advantages over direct calls. First, it is asynchronous: a device that is slow or temporarily unavailable does not block the device sending the message. Second, it is observable: every message through the channel is logged, which means the communication history between devices is always available for debugging.

The inbox is a specific channel pattern: a queue of notifications addressed to the Claude Code session, written by Igor and other devices. At the start of every session, the inbox is checked and any unread notifications are surfaced. High-urgency notifications (like a scope guard block or a safe-mode trip) surface immediately. This is the mechanism by which the human-AI collaboration stays synchronized even when the human has been away.

## Compiled Inference in the Platform

The platform itself is an application of the compiled inference principle.

The early versions of the system had logic spread across many components — Igor contained web server logic, logging utilities, channel management, and queue handling alongside its cognition code. This was the blank-canvas problem at the architectural level: every concern mixed with every other concern.

The rack model separates those concerns. Igor contains cognition. Librarian contains knowledge management. The web server contains HTTP handling. Each device does one thing. When something needs to change in one area, only the relevant device changes.

The Scraps device is the most direct application: it holds the deterministic scripts that have been compiled out of Igor's inference pipeline. When a pattern in Igor's reasoning was stable enough to encode as a script — an IMAP lookup, a standard query format, a known data transformation — that pattern moved from "something Igor reasons about" to "something Igor calls." The reasoning was done once and compiled. The cost of future executions dropped to near zero.

## Starting From Zero

One of the design requirements of UU is that it should be possible to clone the repository, run one command, and have a working system. No manual configuration, no required setup guide, no "you need to set these environment variables first."

The launcher (`./igor`, `./superclaude`, `./uu`) detects the platform, bootstraps the virtual environment if needed, runs the fresh-install wizard if no instance exists, and launches the appropriate components. Everything about how this works is hidden from the user. The only question the system asks on a clean install is: what do you want to call this instance?

This is a statement of values. A system that requires an expert to set up is a system that only works for experts. A system that works from a clean clone is a system that can be shared, reproduced, and evaluated by anyone. The complexity of the setup is the platform's problem, not the user's.
