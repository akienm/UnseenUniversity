# Chapter 3: Workflow, Audits, and Model Selection

---

## The Shape of a Day

A day of AI-assisted development has a rhythm. Understanding that rhythm — and why it is shaped the way it is — is more useful than any individual technique.

The rhythm has four beats:

1. **Orient** — load current state, review what is in flight, confirm what matters today
2. **Design** — for any new work, produce a written design before writing code
3. **Sprint** — execute against tickets, one at a time, with audits at each gate
4. **Close** — record what shipped, update state, verify hypotheses that have matured

Each beat has structure. The structure is not overhead — it is the mechanism by which work is reproducible, navigable, and improvable. A day without structure produces output. A day with structure produces output *and* compounds the system's ability to produce better output next time.

---

## Orient: Starting From a Known State

The first act of every session is a state load. Not a reconstruction from memory — a read from durable storage.

What goes into the state load:
- Active tickets and their current status
- Recent decisions and their hypotheses
- Any consequence-check tickets that have matured
- Anything the AI needs to know that is not obvious from the code

The state load should take less than a minute and should leave the AI with a clear picture of what is in flight, what is blocked, and what the priority order is. If it takes longer than a minute, the state is not well-organized enough.

The orient step also surfaces alignment: if Igor has been running autonomously and something unexpected happened overnight, the orient step is when that surfaces. The human reviews, confirms direction, and the session begins from a shared understanding rather than an assumption.

---

## Design: Writing It Down Before Building It

Every non-trivial change starts with a written design. The design does not have to be long. It has to answer four questions:

1. What is being built? (one sentence, positive framing — what the system will do, not what it will stop doing)
2. What is out of scope? (explicit boundary prevents scope creep)
3. What files are affected? (catches hallucinated targets before implementation)
4. What does passing look like? (runtime-observable signal, not just "tests pass")

The design goes through an audit before tickets are filed. The audit checks nine things: goal alignment, observable success condition, alternatives considered, constraints named, closing questions asked, conflicts with recent decisions, compliance with system rules, decomposition into atomic units, and per-unit executor and inertia assignment.

This is not bureaucracy. It is the cheapest place to catch the expensive mistakes. A design audit costs one pass of inference. A production bug caused by a design assumption costs much more.

After the design audit passes, the design is broken into tickets — one ticket per atomic unit of work. Each ticket links back to the decision that spawned it. The link is bidirectional: from ticket to decision, and from decision to all tickets it spawned. This is what makes the work navigable.

---

## The Ticket

A ticket is the unit of work. It has a specific shape:

- **Title** — short, imperative, under 80 characters
- **Size** — S, M, L, or XL, calibrated to a single PR
- **Description** — the problem and proposed shape
- **Affected files** — explicit, not TBD where avoidable
- **Design rules** — which system constraints apply
- **Scope boundary** — what is in and what is out
- **Test plan** — specific tests to run, or explicit "no tests because: reason"
- **Decision ID** — the decision that spawned this ticket

The ticket is not a bureaucratic artifact. It is a specification that the AI can execute against without ambiguity. A well-written ticket eliminates the blank-canvas problem for the sprint: the AI knows exactly what it is building, what files it is touching, and what passing looks like. The variance that comes from ambiguity is compiled out of the sprint before it begins.

Tickets are audited before they go to sprint. The audit checks for duplicates, scope that is too large, missing test plans, and affected files that do not exist. A ticket that fails the audit does not get sprinted — it gets fixed first.

---

## Sprint: The Execution Loop

A sprint is the execution of a single ticket from claim to close. The loop:

1. **Claim** — mark the ticket in-progress
2. **Infrastructure brief** — review the tools, base classes, and proxies that apply to the affected files; confirm the plan uses preferred forms
3. **Pull** — get a clean base before making changes
4. **Build** — write the change, tests alongside
5. **Cleanup** — review the diff; remove debug prints, commented-out code, unused imports
6. **Test** — run the test suite; no committing with failing tests
7. **Commit** — stage specific files by name, never by wildcard
8. **Close** — record what actually shipped in one sentence

The cleanup step is not optional. The diff review before staging is the last chance to catch debris before it becomes part of the permanent record. A clean diff is the signal that the sprint is ready to ship.

The test step is not optional. Tests exist to verify that the change did what the ticket said it would do, and did not break what it was not supposed to touch. A green test run is a precondition for commit, not a best-effort check.

## The Audit Pyramid

Audits run at multiple layers, each targeting a different class of failure at the cheapest model that can catch it.

**Design audit** — runs before tickets are filed. Checks goal alignment, observable success, alternatives, constraints, closing questions, recent conflicts, system rules, decomposition, and executor assignment. Catches wrong-direction work before any code is written.

**Ticket audit** — runs before each ticket goes to sprint. Checks for duplicates, scope fit, affected files, test plan completeness. Catches specification problems before they become implementation problems.

**Pre-code audit** — runs at the start of each sprint, after the plan is formed. Checks that the affected files exist, the symbols exist, and the approach uses preferred infrastructure. Catches hallucinated targets and wrong-tool choices before any edits are made.

**Post-build audit** — runs after implementation, before commit. Reviews the diff for debris, scope drift, and missing test coverage. Catches implementation drift from the design.

**Consequence audit** — runs weeks after a significant change ships. Reviews the original hypothesis against what actually happened. Catches silent regressions and unintended effects that only appear at scale.

**Day-close audit** — runs at end of day. Checks test suite health, dead code, registry coherence, dependency hygiene, TWM coverage, credential exposure, and a handful of registered one-off checks. The maintenance heartbeat of the system.

Each layer uses the cheapest model that can reliably catch the failures at that layer. Mechanical checks (file exists, symbol exists) run on lightweight models. Design coherence checks run on capable models. The audit pyramid is itself an application of compiled inference.

## Model Selection in Practice

The escalation path is explicit. Every operation starts at the cheapest tier that can handle it.

**Lightweight model, low temperature** — mechanical tasks: rename a variable, add a log call, generate a test stub from a known pattern, run a checklist audit with defined pass/fail criteria. Low variance, high repeatability. Fast and cheap.

**Mid-tier model** — tasks requiring synthesis: write a function implementing a specified interface, review a design for coherence, debug a failing test, draft a ticket from a decision. More judgment required, more tokens, higher cost. Still not the most expensive option.

**Capable model** — judgment-intensive work: design audits requiring evaluation of tradeoffs, novel problems with no established pattern, decisions affecting high-inertia parts of the system, consequence audits requiring interpretation of ambiguous evidence. Reserved for work where the cheaper models have demonstrably insufficient judgment.

The practical rule: start at the bottom, escalate explicitly when the output quality from the cheaper tier is not sufficient. Do not default to the most capable model because it is more comfortable. The comfort is an illusion — the capable model on a blank canvas produces more variance than the lightweight model on a well-compiled skill.

Temperature is the other axis. High temperature produces creative, varied output — useful for exploration and brainstorming. Low temperature produces focused, deterministic output — useful for execution and checklist work. Match temperature to task type. Most implementation work benefits from lower temperature. Most design exploration benefits from higher temperature.

## Closing the Loop

At the end of the day, the close step records what shipped and updates the durable state. Tickets are marked done with a one-sentence summary of what actually shipped. Hypotheses that have matured are reviewed against evidence. Consequence tickets with passed gate dates are promoted to actionable.

The close step also asks: what would have been better? Not a retrospective — a short improvement reflection. What patterns emerged today? What was the biggest friction point? What one change would have the highest impact on tomorrow's work? The answers feed the system's improvement loop.

This is how the system improves itself. Not by accumulating features, but by progressively compiling what works and discarding what does not.
