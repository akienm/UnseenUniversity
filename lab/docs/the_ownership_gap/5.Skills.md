# Chapter 4: Skills as Built

---

## What a Skill Actually Is

A skill is a compiled checklist. It takes a class of work that would otherwise require the AI to reason from scratch and encodes the approach as a specific sequence of steps.

The difference between a skill and a prompt is precision. A prompt says "do this thing." A skill says "do this thing, in this order, checking these conditions, using these tools, with these specific gate conditions that block progress if not met." The skill removes the blank-canvas problem for a defined category of work.

Skills are not static. They are updated whenever the AI gets something wrong — whenever a step is missed, a tool is used incorrectly, or an output does not match expectations. The skill is the accumulated learning of every previous execution of that class of work. It is compiled inference applied to process.

## The Core Skills and Why They Are Shaped the Way They Are

### /decided (formerly /sorted)

The closing mark of a design conversation. When a design discussion reaches a conclusion, `/decided` makes it durable: extracts a testable hypothesis, runs the design audit, drafts tickets, runs the ticket audit on each draft, files the batch, writes the decision record to the memory palace, and appends to the decisions log.

Why it is shaped this way: the design step is where the most expensive mistakes are made and the cheapest to fix. Forcing hypothesis extraction before ticket filing means every design decision has a verifiable outcome attached. Forcing the audit before filing means specification problems are caught at design time, not sprint time.

The skill is named `/decided` (not `/design`) because the name signals completion, not process. By the time you run it, the decision has been made. The skill's job is to make it durable, not to help make it.

### /sprint-ticket

The execution unit for a single ticket. Claim → infrastructure brief → pull → build → cleanup → test → commit → close → savestate.

Why it is shaped this way: each step is a gate. The infrastructure brief prevents the AI from using deprecated patterns. The cleanup step prevents debris from accumulating in the codebase. The test gate prevents broken code from shipping. The specific-file staging prevents accidental inclusion of sensitive files.

The gates are not optional. Each one exists because the failure it prevents has happened, and the cost of the failure was higher than the cost of the gate.

### /sprint-batch

Runs a set of tickets in dependency order. Resolves the target set, topologically sorts by dependencies, runs shared setup once, calls `/sprint-ticket` per ticket, handles failures mid-batch, runs teardown.

Why it is shaped this way: shared setup amortizes overhead; per-ticket execution isolates failure; topo-sort prevents dependency violations. A batch is not a faster version of running tickets one at a time — it is a structured ordering that respects the dependency graph.

### /day-close

The end-of-day shutdown ritual. Savestate → close the slate → run the day-close audit → fix small findings → commit → read the closed slate → push tickets to GitHub → sync docs DB → update DSBs → prune old logs → create GitHub Discussion → post slate → write palace rollup → commit docs → savestate → autocompact.

Why it is shaped this way: the close ritual is the mechanism by which the system's state is durable across sessions. Every step captures something that would otherwise be lost. The audit is mandatory because debt accumulates invisibly and the close is the reliable time to catch it.

### /context-load

The start-of-session state load. Reads today's slate, checks the inbox for Igor notifications, reads recent decisions, confirms what is in flight.

Why it is shaped this way: every session starts from a known state, not from memory reconstruction. The inbox check surfaces time-sensitive notifications from Igor before the human's attention goes elsewhere.

### /audit-design and /audit-ticket

Structured review checklists applied at design time and ticket-filing time respectively. Each check has a positive shape (what to look for), a failure condition, and an AMEND message (what to do when the check fails).

Why they are shaped this way: audits are most valuable when they are specific and actionable. A check that says "is this good?" produces vague output. A check that says "does this name a runtime-observable success condition, not just 'tests pass'?" produces a specific, correctable failure. The audit is a compiled list of the ways things have gone wrong in the past.

### /debug

The autonomous mixed-mode debugger. Takes freeform input — a ticket id, a time, a pasted log window, a symptom description — and produces a structured report without requiring the human to investigate interactively.

The skill runs in phases, cheapest first:

1. **Pointer resolution** — a small inference step to parse the freeform input into a structured query: what time window, which component, what log file, which ticket if any.
2. **Scraps extraction** — deterministic log parsing at near-zero cost: grep the relevant window, extract timestamps, isolate the failure event.
3. **Haiku analysis** — lightweight inference on the extracted log: pattern matching, known failure signatures, surface-level diagnosis.
4. **Sonnet synthesis** — heavier inference only when Haiku cannot produce a confident hypothesis: cross-correlate multiple log sources, reason about state at failure time, generate ranked hypotheses.

The output is a structured report, not a conversation. The report format:

```
SUMMARY:     one sentence
WHERE:       file:line or component:function
WHAT:        what happened
STATE:       relevant variable values or queue state at failure time
LOG WINDOW:  the relevant lines
HYPOTHESIS:  ranked list of probable causes
NEXT:        specific remediation steps
```

Why it is shaped this way: interactive debugging is expensive because each question requires a round trip through inference. The escalation ladder amortizes that cost — most bugs are diagnosed at the Scraps or Haiku level, and the Sonnet call fires only when cheaper steps cannot close. The structured report format means the human reads a conclusion, not a transcript of the investigation. This is compiled inference applied to debugging: the first time a failure pattern is seen, inference is required; after it is encoded in Scraps patterns, it is free.

## How the Skills Integrate

The skills are not independent tools. They form a pipeline:

```
/context-load
  → /decided (design → hypothesis → audit → tickets)
    → /sprint-batch (topo-sort → /sprint-ticket × N)
      → /day-close (audit → docs → sync → savestate)
```

Each skill produces outputs that the next skill consumes. `/decided` produces tickets. `/sprint-batch` consumes tickets and produces commits. `/day-close` consumes the day's work and produces durable state for the next session.

The integration is what makes the system self-improving. Each pass through the pipeline produces not just the immediate output (the shipped feature, the closed ticket) but also an updated model of what works and what does not. The skills are the mechanism by which that learning is encoded and applied.

## The Skill Maintenance Loop

Skills are updated when they produce wrong output. The process:

1. Something goes wrong in a sprint — a step is missed, a tool is used incorrectly, a gate is bypassed
2. The root cause is identified: is this a skill gap (the skill does not cover this case) or an execution failure (the skill covers it but was not followed)?
3. If it is a skill gap, the skill is updated to cover the case
4. The update is specific: one new step, one new gate condition, one new AMEND clause — not a wholesale rewrite
5. The updated skill is used on the next execution of that class of work

The key constraint: skill updates are specific, not sweeping. A skill that is rewritten wholesale loses the accumulated precision of all previous updates. Surgical additions preserve what works while extending coverage.

This is how the system's process layer compiles over time. Each update to a skill is a piece of reasoning that no longer has to be re-derived from scratch. The skill is progressively more complete. Execution is progressively more reliable. The cost per correct outcome falls.

## What the Skills Do Not Cover

Skills cover known, repeatable classes of work. They do not cover:

- Novel architectural decisions (these require the design step, not a skill)
- Ambiguous situations where the right path depends on context the skill cannot encode
- Work that has not been done before and has no established pattern

For these cases, the system escalates to the capable model in freeform mode — but with one constraint: the output of that session should produce either a decision record (if a design decision was made) or a new skill (if a repeatable pattern was discovered). The session is not wasted. It produces compiled output.

This is the core loop: freeform reasoning on novel problems, compiled inference on known patterns, and a continuous process of moving things from the first category to the second.
