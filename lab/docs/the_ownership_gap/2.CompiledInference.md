# Chapter 1: Compiled Inference

---

## The Token Ladder

Here is the shape of the cost problem in concrete terms.

Ask an AI to do something in plain English — no constraints, no prepared tools, just a request. It parses what you want, reasons about how to do it, decides on an approach, executes, interprets the result, and prepares a response. Flexible. Capable. Expensive. Call it roughly 1,000 tokens.

Now write that same task down as a skill: a specific sequence of steps, a checklist the AI follows instead of reasons through. The AI no longer has to figure out the approach — it executes the procedure. Less reasoning, less variance, fewer tokens. Call it roughly 300 tokens.

Take the deterministic parts of that skill — the parts that are always the same regardless of context — and put them in a script. The AI calls the script, reads the result, decides what to do next. Call it roughly 100 tokens.

Each step down this ladder has less ambiguity than the one above it. Each step is cheaper, faster, and more predictable. The ladder does not have a fixed bottom — infrastructure primitives cost almost nothing to invoke. The question for any given operation is: *how far down the ladder has it been compiled?*

## What Gets Compiled

Not everything can be fully compiled, and that is not the goal. The goal is to identify which parts of a process are stable enough to encode — and encode them — so that inference is reserved for the parts that genuinely require it.

A useful heuristic: if you would write the same prompt twice, it belongs in a skill. If the skill runs the same command every time, that command belongs in a script. If the script queries the same data source in the same way, that query belongs in infrastructure.

What remains at the top of the ladder — what genuinely requires freeform inference — is the judgment work: evaluating ambiguous situations, designing solutions to novel problems, deciding what the evidence means. This is also the most valuable work. The compilation process does not eliminate that work. It clears the path to it by removing everything that does not require it.

## The Model Selection Layer

There is a second dimension to the cost problem that the token ladder does not fully capture: not all inference is equally expensive, even at the same token count.

Modern AI systems offer a range of models at very different capability and cost levels. A capable but expensive model (call it Opus) costs roughly ten times as much per token as a mid-tier model (Sonnet), which costs roughly five times as much as a lightweight model (Haiku). The right approach is not to default to the most capable model for everything — it is to start with the cheapest model that can handle the task and escalate only when necessary.

A well-structured AI workflow looks like this: small, routine tasks go to the lightweight model with low temperature — constrained, deterministic, cheap. When a task requires more judgment or synthesis, it escalates to the mid-tier model. The most capable model is reserved for the hardest problems: architecture decisions, audit reviews, novel situations where cheaper models have demonstrably failed.

This is not a compromise on quality. It is the opposite. A lightweight model following a well-compiled skill outperforms an expensive model reasoning from a blank canvas — in cost, in consistency, and usually in correctness, because the skill encodes the patterns that the expensive model would have had to rediscover.

The practical implication: the escalation path should be explicit and built into the process, not left to chance. Every operation should start at the cheapest tier that can handle it, with clear criteria for when to escalate. This is where most organizations leave significant money on the table — not because they are using the wrong models, but because they have no escalation logic at all.

## The Ownership Gap

The creation/ownership gap is the central economic fact of AI development right now.

AI collapsed the cost of creation. That is real and it is not going away. The question is what comes after creation — and the honest answer is that most organizations have not thought carefully about it.

Ownership cost is everything that happens after the first working version: bug fixes, behavior changes, integration updates, performance tuning, onboarding new people to maintain the system, recovering from production incidents. For a traditionally-built system, these costs are roughly predictable. The codebase can be read. The decisions can be traced. The person who built it can explain what it does.

For an AI-built system with no compiled structure — where every session started fresh, where the reasoning evaporated when the context window closed — ownership cost has a different shape. Every maintenance task requires as much inference as the original build, because there is nothing to build on. The reasoning has to be reconstructed from scratch each time. Token costs compound. Variance compounds. The system becomes harder to work with, not easier.

The organizations that are discovering "AI costs more than humans" are discovering the ownership gap. They collapsed creation cost. They did not think about ownership cost. Now they are paying both at once.

## The Feedback Loop

The compilation ladder does not climb itself.

What drives it is a feedback loop: form a hypothesis about what can be compiled, ship the change, measure whether it worked, record the result, refine and repeat. Without this loop, the system drifts. Patterns get discovered and forgotten. Improvements get made and then undone. The organization learns informally — which means it does not really learn at all.

With the loop running, the trajectory is different. Each week, more of the work runs at lower cost. Each month, the organization's AI capabilities are meaningfully more mature than they were the month before. The system knows what it has tried, what has worked, and where the remaining variance is concentrated. Improvement compounds.

The key is that the loop is explicit and tracked. Informal retrospectives do not compound. A record of hypotheses, outcomes, and confirmed patterns does. The infrastructure for this does not have to be elaborate — it has to be consistent.

## Why This Is Not New

Everything described in this chapter is a restatement of something software engineering has known for decades.

Change isolation via layered abstraction is compiled inference applied to test architecture. A standard operating procedure is compiled inference applied to human process. A library function is compiled inference applied to reusable code. Hardware is compiled inference taken all the way down to silicon.

The AI-specific version of the insight is this: the compilation gap has never been shorter. You can use AI to discover, validate, and document a pattern before compiling it — collapsing the time from "someone figured this out" to "the organization runs it reliably and cheaply."

That gap closure is the actual economic opportunity in AI. Not the creation speed, which is real but limited by the ownership problem. The opportunity is to use AI to build the structure that makes AI progressively cheaper to operate.

Every other chapter in this book is an application of that principle to a specific layer of the problem.
