Deterministic AI Development: Levers

A layer on top of skills — Akien Maciain, Test Automation Architect

Models misbehave most when given the most latitude. Skills are like checklists of things to do — they keep the model focused on doing just what you want. But skills alone do not complete the work. They work best as part of a workflow that handles design and implementation.

My experience comes from building akienm/TheIgors, an ongoing research project into using graph trees for reasoning. ~200K lines of code, which works, and has only ever been seen by Claude. I've reviewed maybe a dozen lines — I designed it, not Claude.

The standout feature of my AI experience is LEVERS: ideas, practices, tools, phrases, or questions that provide outsized benefits. In aggregate, they let me treat Claude as a savant — brilliant in some ways, forgetful in others. Rather than fighting that, we build around it: a resilient structure that keeps its forgettings from becoming an issue.

Levers

Keystone patterns that provide outsized returns

1. A Notebook

Mine is a text file. Log files capture all chats. The notebook captures the points or ideas that stood out so I don't have to find them again.

  • For writing if it'll be more than a sentence — too many times I've accidentally cleared Claude's terminal. Compose in an editor, paste in.
  • Blocks of stuff I might paste in more than once.
  • Commands to paste, key moments in the project.
  • Does not have to capture everything — there are other logs for that.

2. My Own Programming and Architectural Experience

Can't be understated. I don't read almost any of the code — but I watch the kinds of bugs that come out of the system, how Claude goes about fixing them, and will often halt and redirect. Key patterns to watch:

  • Claude tends to reuse out-of-date information
  • Claude tends not to centralize concerns to minimize maintenance
  • Claude tends to get lost in the weeds in some circumstances
  • Claude won't remember fixing the same thing yesterday — or last week

3. Process from the Start

Humans have a love/hate relationship with process. Not Claude. As long as it's spelled out in the skill, it'll follow it to the letter. Everything gets designed, designs audited, testing added (TDD, no flaky tests allowed), designs approved, ticketed, audited, sprinted, built, the result audited, planned tests run, ticket closed.

4. Project State Information — Kept Outside the Context

All state lives in Postgres with embeddings. Igor has semantic search internally — Claude can search for any follow-on piece of data after CLAUDE.md for zero tokens. State includes:

  • Project overview
  • Available tools and skills
  • Tickets, epics, slates (echoed to GitHub at day's end)
  • Project documentation tree (architectural .md files in repo, inline for operational notes)
  • Ticket state written into titles: CLOSED:, BLOCKED:, sprint, etc.
  • Decision history

State changes are saved immediately and durably. The model's context is disposable.

5. Tooling and Automation

  • Claude itself is tooling — to development processes
  • Audits — force mistakes and ambiguity out at every layer. See Audit Pyramid.
  • External Tools Internalized — Claude can scrape pages, install libraries, run inline scripts, build its own tooling
  • Logging tree — base class for all objects handles all logging. Logs are forensic. Timestamp from a failure in a chat log leads to the correct point in the master log.
  • MCP — mail service between models. Channel between Claude Code and Igor.
  • Capabilities — wrapping behavior in a named capability is the most certain way to have an LLM do something exactly the way you want and when you want it.

6. Powerful Questions

These bring things into view you couldn't imagine beforehand. Always lay out your own ideas first, then ask:

  • What have I left out?
  • What have I missed?
  • What would be more efficient?
  • What best practices or patterns could we use to good effect?
  • What could we do better?
  • How do other people do this?

7. Rob

One of the first learnings: ask Claude to create "Rob" — a developer so grateful they finally have a QA person on the team. Ride on Rob's shoulder for activities such a developer would have day to day.

8. About Claude

  • Sessions — Even in long sessions, save state constantly. Generic compact+preserve: /compact preserve: Read today's slate: ~/.TheIgors/claudecode/YYYYMMDD.slate.txt. In-flight and Next: see slate.
  • Impatience — Claude will defer things or give fast non-architectural solutions. Point that impatience well. Watch for "I thought we'd implemented that!"
  • Focus — Large contexts aren't infinite. LLMs function well on summaries. Documentation summarizes rules and gives pointers.
  • /advisor — Sonnet can call Opus when uncertain.
  • /loop — Periodic check-ins on something.
  • /schedule — Tell Claude to do a thing at a specific time.
  • --dangerously-skip-permissions — Stop the permission nagging in a project where you don't read the code.
  • Instructions in chat ≠ Instructions in skills. The latter is a checklist; the former is more what ye' might call guidelines…

9. Skills-Based Workflow

My own workflow, built by asking lever questions like "How are the best in the field doing this right now? What's been published? How could we do better?"

10. Hypothesis Extraction & Consequence Checking

Every design decision (/decided) formally extracts a testable hypothesis before tickets are filed. Three mandatory questions:

  1. Which goal does this serve? (G-xxx or "none, reason")
  2. What should be observably different after these tickets ship? (plain English, falsifiable)
  3. How will we know? (metric, log line, behavior, or eval)

For M/L/XL decisions, a gated consequence-check ticket is auto-drafted — predicted unintended effects, signals to watch, and a gate condition (date or observable event). Consequence-checking becomes tracked work, not an informal afterthought.

11. Encapsulation as First-Class Design Principle

Workers request work via encapsulated black-box interfaces; subsystems do not reach across boundaries they shouldn't; callers don't know internals they don't need to. This appears as:

  • Dispatch IS assignment — queue is black box; workers call "anything for me?" periodically
  • Audit-design Check 10 — encapsulation surface detection (subsystems knowing internals they shouldn't)
  • Expert lenses — systems architect + process/meta explicitly scan for coupling that could be a service
  • Pattern extends beyond code to work management itself — Goal 3.7 (shared task service) generalizes for multi-agent platforms

12. Serious Goal Tracking

Goals are formally tracked in the database with G-xxx identifiers. Every /decided links to a G-xxx via testable hypothesis, or explicitly documents "none, reason." Creates a traceable line: goal → hypothesis → decision → ticket → commit.

Workflow

The daily and per-decision flow

/context-load

The agent reads the project overview, list of available tools, palace rules, and today's slate. Gets centered. About 2K tokens.

/design

Work through any tickets that are blocked. Free-form discussion until a point of design is clear enough that Haiku can reliably build it without fail. Last questions always include: "What am I missing? What could we do better?"

/note

Adds any random note that might be important later to the day's slate.

/ticket

Any issue that comes up along the way so we address it later.

/decided

All issues under discussion are resolved enough to go to sprint. Before filing tickets:

  1. Extracts a testable hypothesis (3 questions)
  2. Runs /audit-design (10 positive checks)
  3. Drafts tickets → runs /audit-ticket on each
  4. For M/L/XL: auto-drafts a gated consequence-check ticket
  5. Files everything via cc_queue.py
  6. Writes decision to palace + decisions log

/sprint / /sprint-batch

Sprint a ticket or sprint a large batch. Sprint-batch topo-sorts by dependencies. Per-ticket: capability-check → verify in_progress status (dispatch IS assignment, no claim) → infrastructure brief → pull+work → cleanup → test → commit+push → close → /savestate.

/audit-precode

Runs automatically between plan approval and the first edit. Validates that named files exist, symbols exist, preferred-paths rules are satisfied, test plan named. Haiku-speed; escalates to Sonnet on high-inertia touches.

/fixit

Shorthand for /decided + /sprint-batch in one go.

/day-close

Cleans up, runs /audit-day, closes out the day's slate. Creates GitHub Discussion, pushes docs.

The Audit Pyramid

Scoped audits, each targeting a different failure class at the cheapest model that can reliably catch it

Tracking

Goals, decisions, tickets, slates, architecture, palace

Goals

Formally tracked in the database using goal_adopt, goal_scan, and goal_close tools, with G-xxx identifiers. Goals persist across sessions. Every /decided links to a G-xxx via testable hypothesis, or explicitly documents "none, reason." Creates a traceable line: goal → hypothesis → decision → ticket → commit.

Ideas

One text file per idea in a folder. Brainstorm and chat with free AIs in web chats to sort details with no paid tokens.

Decisions

/design starts a design session. Free-form discussion until a point of design is clear enough that Haiku can reliably build it without fail. The last questions are always: "What am I missing? What could we do better?"

When done, /decided. Before filing, it extracts a testable hypothesis with three mandatory questions.

Slate

A slate is a day's work. Sections:

  • In-flight — what are we doing right now
  • Planned — what's still planned for this slate
  • Ad hoc — reactive additions
  • Done today

When done: /day-close.

Architecture & Code

Everything documented at a high level in the project in .md files in the repo. Actual documentation for each file is IN THE FILE at the top. The AI reads architecture files first, then file tops, then functional code — minimizing the number of files to look at.

Memory Palace

A structured Postgres tree (theigors/rules/*, theigors/decisions/*, theigors/audits/*, etc.) that serves as the canonical index for conventions, rules, audit telemetry, and decision history. CLAUDE.md is a thin bootstrap shim; the palace is the source of truth.

Palace nodes can be read individually (memory_get), searched (memory_search), or bulk-loaded by type. The model spends zero tokens re-deriving conventions it already decided.

Preferred Paths

A palace subtree (theigors/rules/preferred_paths/*) cataloging deprecated patterns alongside preferred replacements — e.g. raw psql calls vs. the MCP proxy, print() vs. IgorBase logger. A scan tool watches 60 days of git history for regressions and surfaces candidates for review, never auto-filing.

Levers Doc Sync Rule

Whenever a skill file is edited or a new workflow capability is added, this document must be updated before the sprint closes (sprint-ticket step 12.5). A forever audit check (levers-doc-skill-sync) fires LOW when skill files were modified in the last day without a corresponding levers doc update.

Why: Skills encode behavior; this doc encodes the design intent behind the behavior. When they diverge, new sessions have no way to understand why the system works the way it does.
