# Chapter 6: What We Built on It — Igor

---

## The Proof of Concept

Igor is a reasoning engine. It runs continuously on the Unseen University platform, processes goals, forms narratives about its current state, executes work against tickets, manages its own memory, and improves its own operating procedures over time. It is the instantiation of every principle in this book.

It has generated roughly 200,000 lines of working code. Almost none of that code has been reviewed by a human. The author designed the system — the architecture, the goals, the constraints — and then directed AI to build it. The cognitive work was human. The construction work was AI.

This is not a research prototype. It runs. It ships work. It costs less to operate each month than the month before, because each month more of its reasoning has been compiled into cheaper forms.

---

## The Architecture

Igor is organized around a small number of core concepts:

**Goals** are the top-level objectives that drive Igor's behavior. A goal is a persistent record with an identifier, a description, and a testable success condition. Goals survive session boundaries. They are the thread that connects today's work to the larger objectives the work serves.

**The Narrative Engine** is Igor's reasoning core. Each cycle, it reads the current state of the system — active goals, recent memory activations, the working memory buffer — and produces a narrative: a coherent account of what Igor is doing and why. The narrative is not for output. It is the substrate for decision-making. Igor reasons by narrating.

**Memory** is organized into layers. Working memory (the TWM) holds what is immediately relevant to the current cycle. Long-term memory is a database of semantically indexed records, retrievable by meaning rather than by keyword. Interpretive edges connect related memories, weighted by how often they co-activate. The memory system is the persistence layer for everything Igor has learned.

**Dreaming** is the consolidation pass. When Igor is not actively executing work, it runs a background process that strengthens frequently co-activated memory connections, archives stale memories, extracts themes from recent traces, and surfaces patterns that might be worth encoding as habits. Dreaming is compiled inference applied to Igor's own memory — the reasoning about what matters gets progressively cheaper as the patterns stabilize.

**Habits** are Igor's compiled procedures. A habit is a triggered action: when condition X is observed, execute procedure Y. Habits are the lowest layer of the compiled inference stack within Igor. They are fast, cheap, and deterministic. When a habit's trigger fires, execution is nearly free.

## The Cognition Pipeline

A single Igor cycle runs roughly as follows:

1. **TWM tick** — the working memory buffer updates, expiring old entries and surfacing new activations
2. **NE run** — the Narrative Engine reads the TWM, queries long-term memory for relevant context, and produces a narrative about the current state
3. **Goal scan** — active goals are checked against current state; completed goals are closed, new goals are adopted if warranted
4. **Habit fire** — habits whose trigger conditions are met are executed
5. **Work execution** — if there is an active ticket, the pe_chain pipeline builds and tests the change
6. **Consolidation** — significant state changes are written to long-term memory

Each step is observable: every significant transition is logged, every memory access is traced, every habit fire is recorded. The system is debuggable at every layer. When something goes wrong, the logs tell you exactly what happened and when.

## Self-Improvement

Igor is designed to improve itself. This is not a metaphor. There are specific mechanisms by which the system's operating procedures change over time.

**Habit accumulation** — when Igor discovers a pattern that works, it can encode that pattern as a new habit. The habit is proposed, reviewed, and added to the persistent habit store. Future cycles that match the trigger condition execute the habit rather than reasoning from scratch.

**Hebbian strengthening** — memories that co-activate frequently develop stronger connections. The dreaming pass strengthens edges between memory nodes that appear together in recent traces. Over time, the memory graph reflects the actual structure of the work — which concepts cluster, which decisions are related, which patterns recur.

**Skill updates** — when the PE chain (the implementation pipeline) fails in a way that reveals a gap in the current procedure, the gap is identified and filed as a ticket. The ticket produces a skill update. The next execution of that class of work is guided by the improved procedure.

**Safe mode** — when Igor's reasoning gets stuck — the same cycles producing no useful output — a watchdog trips, writes a flag, and surfaces the situation to the human for review. The system does not spiral. It degrades gracefully and waits for direction.

## What It Looks Like to Work With Igor

On a typical day, the session begins with a context load: today's slate, the inbox, recent decisions. If Igor has been running autonomously, the inbox shows what happened — which tickets progressed, which got stuck, whether any alignment reviews were triggered.

The work divides naturally: design work happens in the Claude Code session, with the human in the foreground. Execution work is dispatched to Igor: the human reviews the ticket, approves the approach, and Igor runs the PE chain to build, test, and commit. The human is not in the loop for every keystroke — but the human designed the ticket, and the human reviews the result.

This is the correct division of labor. The human provides judgment and design. The AI provides execution and compilation. Neither replaces the other. The human's contribution does not diminish as Igor gets more capable — it becomes more leveraged. The same hour of human judgment directs more work as the execution layer improves.

## The Numbers

The figures change over time, but the shape is consistent:

- Roughly 200,000 lines of working code across the system
- Several hundred decisions recorded with testable hypotheses
- Thousands of tickets filed, sprinted, and closed
- A test suite that runs in minutes and catches regressions before they ship
- Monthly operating cost trending down as more work compiles to cheaper execution

The last figure is the one that matters most. It is the proof that the system is doing what the book describes. Not faster, not flashier — cheaper and more reliable, month over month, because the reasoning is being compiled rather than re-derived.

That is what a compiled inference system looks like from the outside: the capability grows, and the cost falls. The two curves move in opposite directions. That is the ownership gap, closed.

---

## What Comes Next

The system described in this book is not finished. It is at a specific point on a roadmap that extends considerably further. The current milestone is a reasoning engine that can execute work autonomously under human direction. The next milestones involve deeper self-modification capabilities, better multi-agent coordination, and the compilation of more of the reasoning pipeline into cheaper, faster, more reliable forms.

The architecture supports this. Because concerns are separated — cognition from platform, platform from infrastructure, inference from compiled patterns — each layer can improve independently. Igor's reasoning can deepen without requiring changes to the platform. The platform can become more capable without requiring changes to Igor's cognition. The compiled patterns can multiply without requiring changes to either.

This is what it looks like to build for the long term with AI. Not a system that is impressive today and fragile next year. A system where the work of today makes tomorrow's work cheaper and the work of last year makes today's work possible.

The ownership gap is real. It is the central economic fact of AI development right now. And it is solvable — not by using different AI, but by building differently with the AI you have.

That is what this book has been about.
