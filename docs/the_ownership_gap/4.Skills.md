# Chapter 4: Skills as Built

---

## What a Skill Actually Is

A skill is a compiled checklist. It takes a class of work that would otherwise require the AI to reason from scratch and encodes the approach as a specific sequence of steps.

The difference between a skill and a prompt is precision. A prompt says "do this thing." A skill says "do this thing, in this order, checking these conditions, using these tools, with these specific gate conditions that block progress if not met." The skill removes the blank-canvas problem for a defined category of work.

Skills are not static. They are updated whenever the AI gets something wrong — whenever a step is missed, a tool is used incorrectly, or an output does not match expectations. The skill is the accumulated learning of every previous execution of that class of work. It is compiled inference applied to process.

---

## Glossary

Terms used throughout this chapter that may be unfamiliar to new readers.

**Igor** — the cognition device. The AI agent that runs continuously on this system, holds long-term memory, performs dreaming and reasoning cycles, and receives tasks. Think of it as the "always-on" background intelligence that accumulates knowledge between sessions.

**Slate** — the daily working state file. A simple text file that tracks what is in-flight, what is planned for today, and what was done. The equivalent of a whiteboard for the current working session. One file per day.

**Memory Palace** — the structured knowledge base. Decisions, rules, concepts, and goals are stored as a navigable graph. Igor reads and writes to it; CC (Claude Code) queries it at session start. Think of it as the system's long-term memory, organized for retrieval.

**Channel** — the message bus between CC, Igor, and other devices. When CC dispatches a task or Igor sends a notification, it goes through the channel.

**Rack / Device** — the plug-in system for capabilities. Each subsystem (the Scraps log analyzer, the Granny dispatch device, the Librarian retrieval service) runs as an independent "device" on the rack. Devices can be started, stopped, and restarted without affecting the others.

**Ticket** — a unit of work in the queue. Similar to a Jira issue, but with mandatory completion criteria baked into the format. Tickets have structured states: triage → sprint → in_progress → awaiting_validation → closed.

**Decision (D-xxx)** — a recorded design choice with a testable hypothesis attached. Decisions live in the memory palace with a clear "we expect to see X by date Y" statement.

**Goal (G-xxx)** — a high-level measurable outcome that decisions and tickets serve. Goals are the layer above decisions.

**Hypothesis** — the testable claim attached to a decision: "if we do X, we expect observable Y by date Z." The hypothesis is what makes a decision falsifiable rather than just a preference.

**Sprint** — working a single ticket end-to-end: claim, build, test, commit, close.

**CC / Claude Code** — the interactive AI coding assistant. The one you are talking to right now. CC handles design, coding, testing, and ticketing within a session. Igor handles long-term memory and cognition between sessions.

## The Core Workflow: Skills in Pipeline Order

The skills form a production pipeline. The sections below follow that pipeline from session start to session close, then cover the supporting skills that feed into it.

---

### /context-load

The start-of-session state load. Reads today's slate, checks the inbox for Igor notifications, reads recent decisions, confirms what is in flight.

Why it is shaped this way: every session starts from a known state, not from memory reconstruction. The inbox check surfaces time-sensitive notifications from Igor before the human's attention goes elsewhere. Reading the slate first means the session begins with today's priorities already visible rather than re-establishing them from scratch.

---

### /design

Marks the start of a design-mode block. Writes a DESIGN_START marker to the slate and sets a design_mode flag, so that `/sorted` knows exactly where the design conversation began.

Why it is shaped this way: `/sorted` can infer scope retroactively from recent conversation turns, so `/design` is optional. It becomes valuable when a session mixes design and execution work — the explicit bracket tells the system "from this point on we were designing." Without it, `/sorted` estimates the boundary; with it, the boundary is exact.

---

### /sorted (formerly /decided)

The closing mark of a design conversation. When a design discussion reaches a conclusion, `/sorted` makes it durable: extracts a testable hypothesis, runs the design audit, drafts tickets, runs the ticket audit on each draft, files the batch, writes the decision record to the memory palace, and appends to the decisions log.

Why it is shaped this way: the design step is where the most expensive mistakes are made and the cheapest to fix. Forcing hypothesis extraction before ticket filing means every design decision has a verifiable outcome attached. Forcing the audit before filing means specification problems are caught at design time, not sprint time.

Why the name is `/sorted` and not `/decided`: the original name was `/decided`, but it caused a recurring problem — smaller models (and occasionally larger ones) would interpret "decided" as requiring a formal design conversation, and would resist using the skill for non-design closures like "get more tokens" or "let's just rename this file." Not every conversation that reaches a conclusion was a *design*. The British colloquialism "sorted" captures the full range: design conclusions, operational decisions, administrative resolutions, and anything else that needs to be made durable. When something is sorted, it is handled — full stop. The rename was deliberate. See the decision record for the rationale.

Why "sorted" prevents the wrong inference: a model that sees `/decided` tends to ask "was a decision made?" A model that sees `/sorted` tends to ask "is this handled?" That shift in frame is the whole point — closing a block of work, not certifying that it was design-shaped.

---

### /ticket

Creates or updates a ticket from conversation context. `/ticket last` tickets the thing just discussed; `/ticket <description>` files a new ticket from free text.

Every ticket has four required sections: affected files, applicable design rules, scope boundary, and completion criteria. The last one is the most important. A ticket without a machine-verifiable done condition is not ready to file — it is an intention, not a commitment. The completion criteria is what makes the ticket falsifiable, and falsifiability is what separates shipped work from endless in-progress.

Why it runs an audit at filing time: by the time a ticket reaches sprint, the design conversation is over and the cost of a specification problem has gone up. The audit at filing catches "this is already done in the code," "this scope is too large to sprint safely," and "this name collides with an existing ticket" — the cheap fixes that become expensive if they reach the build step.

---

### /sprint

Thin orchestrator: select a ticket (by ID, "last," or next in queue), call `/sprint-ticket` to execute it, then call `/autocompact` to release state. All execution logic lives in `/sprint-ticket`; `/sprint` is only the selector and the post-sprint cleanup wrapper.

Why it is shaped this way: the selector logic and the execution logic are different concerns. The selector needs to understand the queue and the human's intent; the execution logic needs to understand the codebase and the ticket. Separating them means `/sprint-ticket` can be called directly (in batch mode) without re-running the selector logic.

---

### /sprint-batch

Runs a set of tickets in dependency order. Resolves the target set, topologically sorts by dependencies, runs shared setup once, calls `/sprint-ticket` per ticket, handles failures mid-batch, runs teardown.

Why it is shaped this way: shared setup amortizes overhead; per-ticket execution isolates failure; topo-sort prevents dependency violations. A batch is not a faster version of running tickets one at a time — it is a structured ordering that respects the dependency graph.

---

### /sprint-ticket

The execution unit for a single ticket. Claim → re-dispatch check → infrastructure brief → pull → build → cleanup → test → commit → close → savestate.

Why it is shaped this way: each step is a gate. The infrastructure brief prevents the AI from using deprecated patterns. The cleanup step prevents debris from accumulating in the codebase. The test gate prevents broken code from shipping. The specific-file staging prevents accidental inclusion of sensitive files.

The gates are not optional. Each one exists because the failure it prevents has happened, and the cost of the failure was higher than the cost of the gate.

**Advisor escalation (D-sprint-ticket-advisor-escalation-2026-06-04):** Two points where `/sprint-ticket` calls `advisor()` before continuing:
1. **Step 2.5 — re-dispatch check:** When `reset_count > 0` (the ticket has been reset from a prior stalled sprint), call `advisor()` before reading any files or forming a plan. The advisor sees the full context and can redirect the approach before a second failed attempt burns more turns.
2. **Step 8 — third test failure:** When the same tests fail for the third consecutive fix cycle within the session, call `advisor()` before attempting another fix. Three failures signals the approach is wrong, not that the fix just needs to be tried harder.

Why advisor at these points: the cost of a runaway sprint (10+ turns of wrong-direction work) exceeds the cost of one advisor call by an order of magnitude. These two moments are the highest-signal indicators that inference alone will not converge.

---

### /fixit

Fast-path shortcut for bug-shaped reactions. `/fixit` = `/sorted` (implicit, covering the most recent discussion) + `/sprint-batch` on the just-filed tickets. Use when Akien says "fix this," "quick fix," or when a bug is already understood and the solution is short.

Why it is shaped this way: the full design-discuss-/sorted-/sprint-batch loop is appropriate for architectural decisions. For a bug where the cause and fix are both already known, that ceremony adds friction without adding safety. `/fixit` keeps the safety (the filing-time audit still runs, the test gate still runs) while removing the unnecessary ceremony. The distinction between `/fixit` and the full loop is not about rigor — it is about matching the overhead to the complexity of the problem.

---

### /test-fix

Bounded retry loop for test failures. Runs tests, fixes failures, retries up to 3 passes, then escalates to the human. Never retries the same fix twice. Diagnoses before acting.

Why it is bounded: an unbounded test-fix loop is a hallucination amplifier. Without a hard stop, the AI will make increasingly speculative changes to make tests pass, drifting further from the correct fix with each pass. Three attempts is enough to catch the easy cases (a missing import, a wrong assertion) and forces escalation on the hard cases (a design mismatch between the test and the implementation) before the diff becomes unreadable.

---

### /commit

Thin commit cycle for ad-hoc commits outside of `/sprint`. Steps: check auth, run tests, stage specific files, commit, pull, push.

Why it exists separately from `/sprint-ticket`: `/sprint-ticket` wraps commit as one step in a larger lifecycle. `/commit` is for work that doesn't fit a ticket — a documentation edit, a one-line fix discovered mid-session, a manual cleanup. Having a separate skill for this means the commit discipline (test first, specific-file staging, no force-push) is enforced even for untracked work.

---

### /savestate

Flushes in-flight state to the slate. Called after filing a ticket, after closing a ticket, and at session close. Writes what is in-flight, what changed, and what comes next — the minimum to reconstruct session context after compaction or a restart.

Why it is called frequently rather than just at session end: compaction can happen at any point in a long session, and the summary model that fires during compaction does not have access to the conversation detail. If savestate only ran at session end, every compaction event would silently discard in-flight context. The frequent cadence is insurance against compaction.

---

### /autocompact

Fires at the end of a work block — after `/sprint`, after `/sprint-batch`, at the end of `/day-close`. Releases the debug session flag, emits the preserve string, fires `/compact` via tmux.

Why it is block-level rather than per-ticket: compaction pauses the session. Running it after each ticket would break the batch flow. The right granularity is the work block: compact when a coherent unit of work is complete, not after every micro-step.

---

### /day-close

The end-of-day shutdown ritual. Savestate → close the slate → run the day-close audit → fix small findings → commit → read the closed slate → push tickets to GitHub → sync docs DB → update DSBs → prune old logs → create GitHub Discussion → post slate → write palace rollup → commit docs → savestate → autocompact.

Why it is shaped this way: the close ritual is the mechanism by which the system's state is durable across sessions. Every step captures something that would otherwise be lost. The audit is mandatory because debt accumulates invisibly and the close is the reliable time to catch it.

---

## Tracking and Knowledge Skills

These skills manage the layers above tickets: goals, open questions, decisions, reference concepts, and the notes that don't fit any of those.

---

### /goal

Creates, lists, updates, blocks, and retires G-xxx goals — the layer above decisions that anchors all design work to measurable outcomes. Goals live in the memory palace as `palace.goals.*` nodes.

Why goals exist as a separate layer: tickets tell you what was done. Decisions tell you why a choice was made. Goals tell you whether any of it moved the needle. Without goals, a project can close thousands of tickets and never ask whether the system is actually better. Goals are the forcing function for that question: every decision should name the goal it serves, and every goal should have a key result that says "we'll know this worked when we see X."

---

### /outcome

Reviews a decision's hypothesis against observable evidence and records whether it confirmed, falsified, or needs more time. Triggered automatically when the last ticket of a decision closes, or manually at any time.

Why it is a separate skill: the hypothesis review is easy to skip. A closed ticket feels like completion. `/outcome` makes the review a first-class step rather than an optional afterthought. If a hypothesis is never reviewed, the learning loop never closes — the system ships forever without knowing whether it is improving. `/outcome` updates the goal's key results, so goal health reflects actual evidence rather than ticket velocity.

---

### /question

Parking lot for observations and hunches that aren't ready to decide. Creates Q-xxx records in the memory palace so half-formed ideas survive compaction.

Why it exists: design conversations surface observations ("I wonder if the dreaming module is why recall is slow") that are worth tracking but not ready to become decisions. Without a place to put them, they fall into the void between sessions. `/question` gives them a home. Questions can later be promoted to hypotheses (ready to test) or closed if the underlying uncertainty resolves. The skill prevents good observations from being lost just because the timing wasn't right to act on them.

---

### /concept

Collaboratively authors and records architectural reference material. A concept is declarative ("the shape of things") — present-tense, authoritative, no ticket-tracking. Writes to `palace.concepts.*` in the memory palace.

Why concepts are distinct from decisions: a decision records a choice made at a point in time ("we decided to use flat files for the queue"). A concept records permanent structural truth ("the queue is the handoff layer between intent and execution — it is always owned by the human, never claimed autonomously"). Concepts are the reference layer that a cold-read AI uses to get the architecture right without re-explanation. They don't age out; they get updated as the architecture evolves.

---

### /note

Logs a milestone, insight, or decision to notes.log and appends to the slate. For things that matter but don't need a ticket, a decision, or a concept — just a durable record.

Why it exists: the tracking system has strong structure for formal design work. It has almost nothing for the informal layer — the observation made mid-sprint, the benchmark result worth remembering, the "good enough for now, revisit later" that should be written down somewhere. `/note` fills that gap with minimal friction: one call, one timestamped line on disk.

---

### /query-ticket

Surfaces the next available ticket from the queue. Read-only — does not claim, assign, or modify anything. Shows what `/sprint` (no args) would pick up next.

Why read-only: the decision to start a ticket is a human decision. `/query-ticket` is the question "what's next?" without committing to an answer. It is the canonical entry point for "should I sprint?" — separate from the sprint itself so the human can inspect, redirect, or override before work begins.

---

### /opentickets

Lists all open, pending, and sprint-ready tickets grouped by status. Fast queue snapshot with no modifications.

Why it exists separately from `/query-ticket`: `/query-ticket` surfaces one ticket (the next one). `/opentickets` surfaces the whole picture — what is in progress, what is blocked, what is waiting for review. It is the equivalent of looking at the whole board rather than just the top card.

---

### /workflow

Prints a 30-second reference map of the full tracking and workflow system: every skill, the complete stack from goals to tickets, and a "where am I?" guide for sessions after compaction.

Why it exists: the tracking stack has multiple layers (goals → decisions → tickets → sprints) and many skills. After a long gap or a compaction event, the most common disorientation is "which skill do I use right now?" `/workflow` answers that question in one call without requiring the human to remember the full skill taxonomy.

## The Audit Suite

The audit skills form a pyramid. Each layer is more expensive than the one below it, but catches what cheaper layers miss. The design is cheapest-first: a check that fires at design time costs one sentence; the same check at sprint time costs a failed build.

---

### /audit-design

A structured review checklist applied at design time — called by `/sorted` after hypothesis extraction and before ticket drafting. Catches a "decision" that isn't actually decided: vague goals, unobservable success criteria, conflicts with prior decisions, undecomposed scope, missing executor assignment.

Why it is shaped this way: audits are most valuable when they are specific and actionable. A check that says "is this good?" produces vague output. A check that says "does this name a runtime-observable success condition, not just 'tests pass'?" produces a specific, correctable failure. The audit is a compiled list of the ways things have gone wrong in the past.

---

### /audit-ticket

Applied at ticket-filing time. Catches duplicate tickets (already filed or already done in code), tickets too large to sprint safely, scope creep, missing test plans, and HIGH-inertia file touches that require an explicit inline approval.

Why both /audit-design and /audit-ticket exist: design-time and filing-time are different moments in the lifecycle with different failure modes. /audit-design catches "this decision isn't ready to become tickets." /audit-ticket catches "this ticket isn't ready to sprint." They are complementary gates, not redundant ones.

---

### /audit-hypothesis

Fires inside `/sorted` between hypothesis extraction and ticket filing. Five checks: is the hypothesis testable? Is the measurement observable? Does it link to a valid goal? Does it contradict a recently falsified hypothesis? Does it name a time horizon?

Why before ticket filing: a hypothesis that is not testable produces tickets that cannot confirm or falsify anything. The audit catches this at the cheapest moment — before any work is planned. An unobservable hypothesis is not a design decision; it is a preference with extra steps.

---

### /audit-precode

Fires inside `/sprint-ticket` after plan review, before the first file edit. Verifies that every file and symbol in the plan actually exists (catches hallucinated targets), gates HIGH-inertia file touches with an explicit approval step, checks for deprecated patterns, validates the test plan.

Why before the first edit: the cheapest place to catch a wrong plan is before the plan touches the codebase. A hallucinated target (a function that the AI believes exists but does not) costs one sentence here; it costs a failed build and a confused diff later.

---

### /audit-smell

Fires after edits land but before tests run. Catches the failure modes specific to large language model code generation: fix-one-leave-many (fix the named instance, miss the identical one three files over), deprecated-path fallbacks (use the old API because it was in training data), helpful refactors that bloat the diff beyond the ticket scope, bare excepts, silent-return-False patterns, missed base class inheritance, dead code.

Why after edit and before test: tests verify correctness. Smell checks verify *shape* — whether the code is the right kind of code, not just whether it produces the right output. A test suite can pass on code that will cause maintenance problems for the next six months. The smell check is the gate for that class of problem.

---

### /audit-debris

Post-test, pre-commit cleanup check. Finds temp files, debug artifacts, staged runtime paths, .env changes that weren't communicated, log-size growth, test database leftovers, misplaced files, doc rot on changed load-bearing files, and subsystem index drift.

Why it exists: a green test suite does not mean a clean diff. It means the code is correct. It says nothing about the temp files that accumulated during debugging, or the log file that is now 400MB, or the `.env` change that affects production behavior and was never mentioned. `/audit-debris` is the mechanical check for "is the diff actually what you think it is?"

---

### /day-close-audit

Mandatory check during `/day-close`. Covers: temp files, leaked runtime state, dead code, test coverage, file placement, code smells, registry coherence, inertia tag drift, thread hygiene, log sizes, OR burn rate, DB schema drift, duplication, habit health, TWM coverage gaps, dependency hygiene, credential scan, simplification review.

Why it is mandatory and not optional: debt accumulates invisibly during a sprint day. Without a structured close check, the accumulated debt is invisible until it causes a failure. The mandatory audit is what keeps the system's health visible even when no individual sprint day generates an obvious problem.

---

### /audit-day

Runs during `/day-close` after `/day-close-audit`. Adds the temporal layer: cross-day watch-for patterns, fix-one-leave-many sweeps across today's commits, subsystem index vs. reality drift, inertia tag drift, TWM coverage gaps, habit health, and auto-drafting of scan-for-rest tickets for anything that wasn't fixable today.

Why both `/day-close-audit` and `/audit-day` exist: `/day-close-audit` is static checks (this file exists, this test passes). `/audit-day` is temporal checks (this pattern appeared in three commits today, this file hasn't been tested in a week). Different lenses on the same codebase, not redundant coverage.

---

### /audit-goal

Goal-time audit. Seven checks on any G-xxx goal: is the goal measurable? Are key results observable? Is the time horizon stated? Does it link to the project's stated purpose? Does it conflict with active goals? Is success distinguishable from business-as-usual? Does it have a clear owner?

Why goals need auditing: a goal that passes none of these checks is an aspiration, not a goal. The audit forces precision at goal-creation time, which is also the cheapest time — before any decisions, any tickets, any sprints have been built around the vague aspiration.

---

### /audit-expert

Expert-lens codebase review. Eleven specialist perspectives (safety engineer, UX designer, security reviewer, performance engineer, and others) each apply their broadest discipline lens to the codebase. Weekly cadence picks three randomly; monthly runs the full panel.

Why multiple expert lenses rather than one comprehensive check: different disciplines find different things. A performance engineer finds query patterns that a security reviewer walks past. The rotation prevents audit fatigue (the same check finding the same things indefinitely) and surfaces the cross-disciplinary issues that appear only when two lenses overlap.

---

### /audit-audits

Meta-audit. Analyzes structured telemetry from every audit layer over time — recurring smells worth promoting to palace rules, catches that land one layer too late (issues that should have been caught at design time but appeared at smell time), dead checks (checks that never fire), false-positive sweeps, cost-per-finding, and cross-layer coherence.

Why the audit system audits itself: audit drift is the failure mode where the system gradually becomes worse at watching itself. Individual audits can go stale (a check for a pattern that no longer occurs), redundant (two checks catching the same thing at different layers), or missing (a new failure mode not yet encoded). `/audit-audits` is the mechanism by which the audit pyramid stays calibrated.

---

### /deep-audit

Weekly broad-lens codebase review. Runs eleven Haiku specialist agents in parallel, synthesizes with Sonnet, posts ranked findings to the channel. Cadence: weekly.

Why it runs in parallel: the eleven specialists are independent. Running them concurrently rather than sequentially means the full review completes in the time of one specialist pass rather than eleven. The synthesis step is what integrates the findings — the parallel structure is a performance optimization, not an architectural one.

## Diagnostic and Observability Skills

---

### /debug

The autonomous mixed-mode debugger. Takes freeform input — a ticket ID, a time, a pasted log window, a symptom description — and produces a structured report without requiring the human to investigate interactively.

The skill runs in phases, cheapest first:

1. **Pointer resolution** — a small inference step to parse the freeform input into a structured query: what time window, which component, what log file, which ticket if any.
2. **Scraps extraction** — deterministic log parsing at near-zero cost: grep the relevant window, extract timestamps, isolate the failure event.
3. **Haiku analysis** — lightweight inference on the extracted log: pattern matching, known failure signatures, surface-level diagnosis.
4. **Sonnet synthesis** — heavier inference only when Haiku cannot produce a confident hypothesis: cross-correlate multiple log sources, reason about state at failure time, generate ranked hypotheses.

The output is a structured report, not a conversation. The report format:

```
SUMMARY:     one sentence
WHERE:       file:line or component:function
WHAT:        what happened
STATE:       relevant variable values or queue state at failure time
LOG WINDOW:  the relevant lines
HYPOTHESIS:  ranked list of probable causes
NEXT:        specific remediation steps
```

Why it is shaped this way: interactive debugging is expensive because each question requires a round trip through inference. The escalation ladder amortizes that cost — most bugs are diagnosed at the Scraps or Haiku level, and the Sonnet call fires only when cheaper steps cannot close. The structured report format means the human reads a conclusion, not a transcript of the investigation. This is compiled inference applied to debugging: the first time a failure pattern is seen, inference is required; after it is encoded in Scraps patterns, it is free.

---

### /diagnose

Unified device diagnostics for any rack device. Surfaces the device's recent trace log, categorizes the issue (code bug / state corruption / external noise), and suggests one specific fix. Usage: `/diagnose <device-name>` (e.g. `/diagnose igor`).

Why one unified diagnostic entry point rather than device-specific tools: a device-specific diagnostic skill requires knowing which device has the problem before running it. `/diagnose` works from a symptom ("something is wrong") rather than a pre-identified target. The categorization step — is this a code bug or a corrupted state? — is what determines the right fix path, and that categorization requires looking at the trace regardless of which device it is.

---

### /map-igor

Full state snapshot of Igor: palace tree, rules, subsystem index, tickets, slates, decisions, gates, MCP connections, IMAP, channels, inbox, logs, startup reads, DB schema, runtime, code map, and active processes. Produces a JSON file and a one-screen stdout summary. Diff mode (`--since=yesterday`) shows what changed.

Why a map skill rather than checking each system directly: Igor's state is distributed across a dozen subsystems. Checking them individually requires knowing where to look and what healthy looks like for each one. `/map-igor` gives a complete picture in one call and makes it easy to diff "what changed today" from "what is always true" — the diff mode is often the fastest path to "why is this behaving differently than yesterday?"

---

### /readigor

Reads Igor's recent replies from the shared channel. Shows what Igor has been saying, optionally filtered to a specific machine.

Why it exists: Igor's responses appear in the channel, not in the CC terminal. If CC sends Igor a task or query, `/readigor` is how CC reads the response. Without it, the channel is write-only from CC's perspective and Igor's output is invisible until Akien manually checks the channel. `/readigor` closes that loop.

---

### /readinbox

Reads unread notifications from Igor subsystems pushed to CC's inbox. Igor's pe_chain escalations, scope_guard HIGH-inertia blocks, ticket trips, and go-live-when fires all land here.

Why a dedicated inbox rather than channel-read: the channel carries all traffic between all devices. The inbox carries only the notifications that specifically require CC's attention. Reading the channel to find CC-directed notifications is expensive and noisy. The inbox is the filtered view — everything there is addressed to CC and requires a response or acknowledgment.

---

### /dream

Manually triggers Igor's dreaming module via a channel message and polls for the dreaming summary response.

Why dreaming is triggerable manually: Igor's dreaming runs on a schedule (typically overnight). Between schedule runs, it can be useful to trigger a dreaming pass manually — after a large batch of new information arrives, before a major design decision, or to test whether the dreaming module is functioning correctly. `/dream` is the manual trigger for that, with a 30-second poll for the response.

## Periodic Review Skills

---

### /weekly-retro

Friday retrospective. Reviews hypothesis confirmation rate, goal key-result trends, and what changes about next week's priorities. Five minutes, three questions, one screen of output. Writes to `palace.retro.YYYYMMDD`. Called automatically by `/day-close` on Fridays.

Why a separate weekly review exists alongside the daily audit: the daily audit asks "is the code healthy?" The weekly retro asks "are we making the right bets?" These are different questions. A project can have a healthy codebase and declining goal progress, or vice versa. The weekly review is the mechanism by which goal-level drift becomes visible before it becomes irreversible.

---

### /eval-run

Weekly capability snapshot. Asks five behavioral questions about what Igor can actually do — independent of ticket velocity or test counts. Feeds goal key-result progress. Run Fridays as part of day-close, or standalone. Writes to `palace.evals.YYYYMMDD`.

Why a separate capability eval exists alongside tests: tests verify code correctness — that a function returns the right value. Evals verify capability — that the system can actually do the thing it was built to do. "The dreaming module runs without error" is a test result. "Igor can synthesize a new insight from yesterday's dreaming pass and apply it to today's ticket" is an eval result. Capability can regress even when all tests pass.

## Utility and Meta Skills

---

### /sprint-loop

Autonomous queue drain. Runs `/sprint-batch` in a self-rescheduling loop until the queue is empty. Schedules a `ScheduleWakeup` *before* each batch starts — the key invariant that ensures the loop survives a compaction event mid-sprint.

Why the wakeup is scheduled before, not after, the batch: if `/autocompact` fires mid-batch and the wakeup was scheduled at the end, the loop dies silently. Scheduling the wakeup first means that even if the session is compacted partway through, the loop resumes automatically when CC is next idle.

*(See Periodic Review Skills above.)*

### /ADCHelp

Prints the skill cheat-sheet to the console, or writes it to a specified file path. Quick reference for all available skills and their one-line descriptions.

Why it exists: the skill list is long. Having a single command that outputs the full reference without requiring a memory palace query or a search makes it practical to use the cheat-sheet during any session, from any working directory.

---

### /export-chat

Dumps the current CC session transcript to a dated markdown file. Recovery snapshot for sessions where the chat history may scroll off the top or be needed for reference.

Why it exists: CC sessions can span hours and hundreds of turns. The in-session context window has a limit. `/export-chat` creates a durable disk copy before compaction or session close removes context that may be needed later — for audit, for recovery, or for sharing.

### /skills-sync

Compares and syncs skills between the master repository and `~/.claude/skills/` on the current machine. Commands: `status` (show managed vs. local-only), `diff` (show content differences), `deploy` (copy master → local), `copy-to-repo` (promote a local skill back to master).

Why a dedicated sync skill rather than manual rsync: the skill set evolves. New skills are added centrally and need to reach all machines. Local-only skills (written on one machine) need a path to promotion. `/skills-sync` makes both directions explicit and auditable, rather than relying on ad-hoc file copying that can leave machines inconsistent.

## How the Skills Integrate

The skills are not independent tools. They form a pipeline:

```
/context-load
  → /design (optional bracket)
    → /sorted (decision → hypothesis → audit → tickets)
      → /sprint-batch (topo-sort → /sprint-ticket × N)
        → /day-close (audit → docs → sync → savestate)
```

Each skill produces outputs that the next skill consumes. `/sorted` produces tickets. `/sprint-batch` consumes tickets and produces commits. `/day-close` consumes the day's work and produces durable state for the next session.

The integration is what makes the system self-improving. Each pass through the pipeline produces not just the immediate output (the shipped feature, the closed ticket) but also an updated model of what works and what does not. The skills are the mechanism by which that learning is encoded and applied.

## The Skill Maintenance Loop

Skills are updated when they produce wrong output. The process:

1. Something goes wrong in a sprint — a step is missed, a tool is used incorrectly, a gate is bypassed
2. The root cause is identified: is this a skill gap (the skill does not cover this case) or an execution failure (the skill covers it but was not followed)?
3. If it is a skill gap, the skill is updated to cover the case
4. The update is specific: one new step, one new gate condition, one new AMEND clause — not a wholesale rewrite
5. The updated skill is used on the next execution of that class of work

The key constraint: skill updates are specific, not sweeping. A skill that is rewritten wholesale loses the accumulated precision of all previous updates. Surgical additions preserve what works while extending coverage.

This is how the system's process layer compiles over time. Each update to a skill is a piece of reasoning that no longer has to be re-derived from scratch. The skill is progressively more complete. Execution is progressively more reliable. The cost per correct outcome falls.

---

## What the Skills Do Not Cover

Skills cover known, repeatable classes of work. They do not cover:

- Novel architectural decisions (these require the design step, not a skill)
- Ambiguous situations where the right path depends on context the skill cannot encode
- Work that has not been done before and has no established pattern

For these cases, the system escalates to the capable model in freeform mode — but with one constraint: the output of that session should produce either a decision record (if a design decision was made) or a new skill (if a repeatable pattern was discovered). The session is not wasted. It produces compiled output.

This is the core loop: freeform reasoning on novel problems, compiled inference on known patterns, and a continuous process of moving things from the first category to the second.
