# Chapter 2: Levers

---

## What a Lever Is

A lever is a practice, tool, question, or idea that produces outsized results relative to the effort it takes to apply. Not all improvements are equal. Some changes to how you work with AI have marginal effects. Others change the trajectory.

The patterns in this chapter are the ones that change the trajectory. They are drawn from direct experience building a large AI-assisted system over several years — roughly 200,000 lines of working code, designed by the author, written almost entirely by AI. The levers are not theoretical. Each one is something that, once found, made the work qualitatively different.

They are organized into three categories:

- **Environment levers** — set up once, pay dividends indefinitely
- **Process levers** — applied repeatedly as part of daily work
- **Mindset levers** — the mental models that make the other two coherent

## Environment Levers

### State lives outside the context window

The context window is working memory. It is finite, it empties when the session ends, and the AI treats everything in it as equally relevant regardless of when it was written. Relying on the context window for project state is the source of a large class of failures: the AI acts on stale information, loses track of decisions made earlier in the conversation, and reinvents things it already built.

The countermeasure is to put state where it persists: a database, a file, a structured record that survives the end of the session. Design decisions belong in a decision log. Ticket state belongs in a queue. Project context belongs in a memory system the AI can query at the start of each session. The context window is for immediate work; everything else is infrastructure.

This is not a workaround for AI limitations. It is the same architecture that makes human organizations function — externalizing knowledge so that work does not depend on any one person's working memory.

### The logging tree

Every component in the system should log at least all its significant state transitions. Not for debugging convenience — for forensic capability. When something goes wrong, you want to be able to trace the failure through the logs to its root cause without reconstructing the context from scratch.

The logging tree means that every component's logs connect to a shared timeline. A timestamp from a failure in a chat session leads back to the corresponding point in the system log, which leads to the component log, which shows the state at the time of the failure. This is the difference between a system that is debuggable and one that requires a rebuild to diagnose.

Build the logging infrastructure first, before it is needed. The cost of adding logging after the fact is much higher than the cost of building it in from the start.

### Preferred paths and capability declarations

AI defaults to the path of least resistance in its training data. If there is a tool the system should use and a general-purpose alternative that is more familiar, the AI will use the general-purpose alternative unless explicitly directed otherwise.

The solution is to declare the preferred path as a capability — a named operation with a specific interface — and put the instruction to use it where it will actually be read. A capability declaration in the system prompt is more reliable than a note buried in documentation. An explicit "use X instead of Y for this operation" instruction in the active context is more reliable than a capability declaration the AI has to remember from three sessions ago.

1. Explicit "Use x and not y when doing z", "Use MCP and not bash for database access"
2. Explicit 'why' on design decisions, tickets, rules, tool directives. So the above becomes: "Use MCP and not bash for database access because we get free logging"

The importance the 'Why' gives to the AI cannot be understated. And sometimes it even helps catch errors. The AI will ask about conflicting whys to be sure.

Constrain the AI to the right path. Do not rely on it to choose the right path independently.

### Searchable memory

Beyond session state, there is a class of knowledge that accumulates slowly and is needed in fragments — design decisions, architectural constraints, the reason a particular approach was chosen over alternatives. This knowledge does not belong in the context window (too much of it, most of it irrelevant to any given task) and it does not belong in one flat file (too hard to retrieve the right piece at the right time).

A searchable memory system makes this knowledge retrievable. The AI can query it at the start of a task, find the relevant precedents and constraints, and work within them without requiring the human to remember and re-state everything. This is the infrastructure for organizational learning. Without it, the organization learns informally. With it, what was figured out last month is available next month without the overhead of re-derivation.

The realization matters less than the searchability, and the lever has a lesson hidden in it. This system started with a database-backed "memory palace" for its dev-process knowledge — decisions, tickets, rules, design notes. Over time, that store was replaced by something simpler: a tree of grep-able JSON files, one fact per file, searched with the same `grep` every developer already knows. The simpler substrate won. The cognition layer — the agent's own semantically-indexed long-term memory, where retrieval-by-meaning is load-bearing — stayed in a database, because there the structure earns its cost. The principle is the same one that runs through this whole book: use the cheapest substrate that actually does the job, and do not pay for structure a task does not need.

## Process Levers

### Design before code

The most expensive mistakes happen before a line of code is written. A wrong assumption about what to build, a misunderstood constraint, a design that does not account for an obvious edge case — these are cheap to fix at design time and expensive to fix after implementation.

AI can generate code very quickly. That speed creates pressure to skip the design step. Resist it. Formal design — even a short one — forces the key questions to be answered before they become implementation bugs. The design also becomes a reference point: when the implementation drifts from the design, the drift is visible and correctable.

The design step does not have to be elaborate. It has to be explicit. A written statement of what is being built, what is out of scope, and what the passing condition looks like is enough to catch most of the expensive mistakes.

### Audits at every layer

An audit is a structured review against a checklist of known failure modes. The power of audits in an AI context is that AI does not resist process — it follows the checklist without complaint, every time, at whatever layer you ask it to check.

The layers matter. A design audit catches problems before implementation. A pre-implementation file audit catches hallucinated file paths and symbol names before the AI starts editing the wrong things. A post-implementation audit catches drift from the design. A consequence audit, run weeks after a change ships, catches unintended effects that only appear at scale.

Each audit layer targets a different failure class at the cheapest model that can catch it. Routine mechanical checks (does this file exist, does this symbol exist) go to lightweight models. Design coherence reviews go to more capable models. The audit pyramid is itself an application of compiled inference: expensive judgment reserved for expensive problems, cheap pattern-matching handled cheaply.

### Hypothesis extraction

Every significant design decision should have a testable hypothesis attached to it before work begins. Three questions:

1. Which goal does this serve?
2. What should be observably different after this ships?
3. How will we know — what is the signal we will check?

The hypothesis is not a formality. It is the mechanism by which improvement compounds. Without it, you can ship a change and not know whether it worked. With it, you have a specific observable outcome to check, a record of what you predicted, and a basis for updating your model of the system.

At the end of the sprint, you check the hypothesis. Confirmed, falsified, or needs more time. The record accumulates. The organization learns which patterns work, which do not, and what to try next. This is what separates informal improvement from structured improvement — and structured improvement is what compounds.

### Consequence checking

Large changes have unintended effects. This is not a failure of planning — it is the nature of complex systems. The countermeasure is not to plan more carefully; it is to explicitly schedule a check.

When a significant change ships, file a consequence-check ticket gated to a future date. The ticket asks: what did we predict would be different, what actually changed, and did anything break silently? This forces the feedback loop to close. It catches the cases where the immediate tests passed but something adjacent degraded over the following weeks.

Consequence checking is not overhead. It is the mechanism by which the feedback loop produces reliable signal rather than wishful signal.

### Ticket everything

Work that is not tracked does not compound. When every change has a ticket — with a clear scope, a test plan, and a link to the decision that spawned it — the work is navigable. You can see what was done, why, and what the expected outcome was. You can trace a behavior change back to the decision that caused it. You can identify which decisions consistently produce clean outcomes and which ones consistently require rework.

A ticket is not bureaucracy. It is the unit of organizational memory for work. Without tickets, the work disappears into the git log. With tickets, the work is navigable from any direction: from the decision that spawned it, from the code it changed, from the outcome it produced.

### Recall first

Every piece of information the system derives through reasoning or lookup has a cost: the inference call, the bash command, the grep across the codebase. When that information has been derived before and stored, re-deriving it is pure waste.

The lever is the lookup order. When a task requires information about the system, ask in this sequence: what do I already know about this? — what can I retrieve from memory? — what do I need to derive fresh? Most of the time, the answer is already stored. The bash command or grep or inference call is the fallback for the case where it is not.

This is the compiled inference principle applied to everyday work. Inference fires once — at the point where new knowledge is first derived — and the result is stored. Future queries hit the stored result at near-zero cost. Over time, a system that applies recall-first accumulates a base of pre-computed knowledge that makes it progressively cheaper to operate. A system that skips recall and re-derives on each query pays the full cost every time.

The practical discipline: before any lookup or reasoning task, ask "what do I know about X?" If the answer is in memory, use it. If it is not, derive it — and then store it so the next time is free.

## Mindset Levers

### Design around the tool

AI has real strengths and real weaknesses. Designing around both — rather than fighting either — is the single most important mindset shift for getting leverage from AI.

The weaknesses are consistent and predictable: AI loses the big picture under pressure, reuses stale information when newer information is not prominent enough, defers architectural decisions when given room to defer them, forgets what it decided last week, and defaults to familiar patterns even when better ones have been established.

Each of these weaknesses has a countermeasure. Big-picture loss → context anchored in durable external state. Stale information → explicit recency signals in the prompt. Architectural deferral → required design step before implementation. Forgetting previous decisions → decision log in the memory system. Familiar-pattern default → explicit capability declarations constraining the choice.

The pattern is always the same: identify the failure mode, build a countermeasure into the environment or process, measure whether it works, adjust. Design around the tool. Do not fight it and do not pretend the weaknesses do not exist.

### Powerful questions

Before finalizing any design, ask:

- What have I left out?
- What have I missed?
- What could be more efficient?
- What best practices could we use to good effect?
- What would be better?
- How do others solve this?

These questions are not rhetorical. Ask them to the AI, after laying out your own thinking first. They reliably surface things that were invisible from inside the design. The cost is one additional pass. The yield is consistent — gaps found at design time rather than in production.

We do build these questions into our skills as well.

### The context window is disposable

Treat the context window as working memory, not as a record. When it fills, compact it and continue from the durable state. Design the work so that no important information exists only in the context window at any point.

This mindset change has a practical implication: the start of every session should be a state-load from durable storage, not a reconstruction from memory. The AI reads the current state of the project — the active tickets, the recent decisions, the relevant constraints — and begins work from a known baseline. The session is not a continuation of the previous session's context. It is a fresh execution against a persistent state.

This is not a limitation to work around. It is the correct architecture. Stateless execution against persistent storage is more reliable than stateful sessions that accumulate drift.

### The savant model

Think of AI as a brilliant but forgetful collaborator. It has extraordinary capability within a session and within its training. It has no persistent memory and will repeat mistakes it made last week unless the countermeasures are in place.

The correct response to this is not frustration. It is architecture. Build the structures that compensate for the forgetting: the searchable memory store, the decision log, the hypothesis record, the durable ticket queue. Then direct the AI's capabilities at the work that requires them, inside the constraints those structures provide.

A savant working inside a well-designed structure outperforms a generalist working on a blank canvas. The structure is the leverage.
